Summary & goal Findings

Scope of work

This covers a full rebuild of our Figma design system, which underpins three product areas: the Presentation Website, the Employee Web App, and the Management Dashboard.

The goal

We want a properly structured, Variable-based design system where every component has a clear property structure, a documented purpose, and a build pattern that scales. Some parts of the file, and some of the visual design itself, could also see a refresh — that's a separate decision from this scope of work, and isn't included in this budget.

What we found

These are general findings from an initial pass, not an exhaustive component-by-component review — that level of detail is what your own file exploration, before this engagement starts, will surface. Ordered from the most significant issue to things already working well.

Foundational

Styles, not Variables

Colour, spacing, radius, and grid are built with Styles rather than Variables, so there's no single source of truth and no way to support fast, systemic updates.

Structural

Component structure is inconsistent

Some components shape the frame itself directly, adding radius and fill to it; others nest a rectangle inside the frame for the same purpose. Both patterns sit side by side in the same file.

Mixed

Properties are inconsistent, too

Some component families are properly built with Properties, others aren't. Cards are a good example: they only have Variants, no Properties, so using one means checking the different variants and guessing which applies where.

Needs restructuring

Some components aren't structurally optimised

Buttons have properly built states and properties, but Primary, Secondary, and Tertiary exist as three separate components rather than a single Property within one Button component. That adds unnecessary duplication, and it's one of the first things to fix when this gets rebuilt.

Untracked

Some colours aren't linked to any style

A number of colour values in the file are hardcoded rather than linked to a Style at all. Even after moving from Styles to Variables, as planned, these need to be found and linked manually first, which makes that migration more work than a straightforward swap.

Partially built

Some components are built but not used within the templates

A few pieces of the system already exist but don't show up anywhere in the actual product screens, which makes those templates less useful in practice.

Needs rebuild

Management Dashboard exists only in wireframe shape

Management Dashboard has an early wireframe pass but no built components yet, so it needs full design and rebuild as part of this scope, not just a refresh. Presentation Website and Employee Web App are further along, though inconsistently: some workflow stages (e.g. Ready for Dev) are populated for one and empty for the other.

By design

The extra numbered pages aren't clutter

What looks like dozens of unlabelled, numbered pages sitting alongside the real structure is intentional: they're a way of adding more pages inside the same file, rather than paying for additional Figma files or seats. Worth knowing when navigating the file, but not something to clean up. That page-vs-file trade-off changes once the multi-file setup goes live (see FAQs) — this describes the current file, not the target structure.

Not a system

The file is generally not annotated

There's no consistent way for components to explain when or how to use them, which makes onboarding, or coming back to the file later, harder than it needs to be.

Good shape

Text styles

In good shape. Most of what was reviewed is correctly linked to text styles.

Strategy

This is not a redesign. It's building the design system properly and applying it everywhere: clean components with one source of truth, pages built from them with Auto Layout, cleanly named layers. The strategy is:

File exploration, on your own time — before this engagement starts, you explore the current Figma file yourself and decide whether you're happy to take this on. This isn't billed, and it isn't a written audit or a set of redesign recommendations — just enough digging to know what you'd be signing up for.
Full component list, before kickoff — you send us a complete list of every component you'll build. We confirm scope together against this list before Milestone 1 starts. Everywhere this brief says "the agreed list," this is what it means.
Build in Figma — the rebuild follows an atomic design structure, working from the smallest unit up: Foundations (Variables, colour, typography, spacing, elevation, grid), Atoms (buttons, inputs, badges, icons), Components (cards, modals, navigation, forms), Templates (page layouts), and Pages (product-ready screens). The file structure reflects this — fewer pages, clearly named, easier to navigate day to day. Full detail on how components need to be built lives in the Designer Instructions tab. AI handles the mechanical parts: setting up Variable collections, migrating existing Styles, scaffolding property structures, and batch operations across the file. The design decisions — what each component looks like and how it behaves — are made by hand.

Each site gets its own file, with the design system built as a separate shared library all three plug into — see the FAQs for the details.
Weekly check-ins (1h) — a standing weekly meeting where you walk through how things are going: what's been worked on, what's been found, progress against the plan. Presented in a way that's digestible for both the product strategist and the founders, since we're coming at this with different levels of design/technical understanding — pace and depth are things we'll figure out together rather than assume. This time is billed separately from the $1,800 fixed fee; the rate gets discussed and agreed before the engagement starts. We don't expect these to be free.
Checkpoint after the pilot — once the first 2-3 components are built in Milestone 1, we pause. You make sure that page meets the standard set out in the Designer Instructions, we test it, and only once it checks out does the rest of the build continue into Milestone 2.

How AI will be used

Given how much restructuring this file needs, AI is used deliberately, and only where it speeds up mechanical work. It is never used in place of design judgment.

AI will

Organise the file — pages, sections, and naming conventions.
Build Variable collections — colour, spacing, radius, shadow/elevation, grid, and migrate existing Styles into them via a direct Figma MCP connection.
Scaffold component properties — boolean, instance-swap, and text properties, once a component's structure has been specified.
Handle batch operations — renaming layers, applying styles/variables consistently across dozens of instances, restructuring pages.

AI will not

Evaluate the file — reviewing structure and scalability requires design judgment that AI review can't substitute for.
Decide what gets built or how it's structured — a decision made together with us, not left to AI. This is not a redesign, so there's no "keep vs. redesign" call to make — just what gets built, matching the Designer Instructions.
Build components visually — given how inconsistent the current patterns are, turning them into one clean component is manual design work.
Write annotation content — usage guidance requires understanding our product and use cases.

Foundation setup

Do these first, before any component work.

Every color, spacing value, radius, font size, and shadow lives as a Figma variable. No raw hex, no ad-hoc pixel values inside components. If you feel tempted to type #FF6B6B directly, stop and add a variable.

Structure variables in two collections:

  • Primitives — raw scale values (color/blue/500, space/4, radius/md, font/size/16)
  • Semantic — meaning-based, references primitives (color/bg/surface, color/text/primary, space/gap/card, radius/button)
Rule: components use only semantic variables. Primitives exist to define semantic ones.

Variable names use forward slashes for hierarchy. color/bg/surface, not Color - BG - Surface or colorBgSurface. This exact naming becomes the CSS variable and JSON key the developer uses. Case-sensitive, hyphen for word breaks (surface-hover, not surfaceHover).

Component names match code component names exactly. If it's a Button in Figma, it's Button in code. Property names too: variant, size, disabled. Values: primary, secondary, ghost, sm, md, lg — exact case, exact spelling.

Prefer boolean and instance-swap properties over variants wherever the states aren't mutually exclusive:

  • disabled, loading → boolean properties
  • iconLeft, iconRight → instance-swap properties
  • label → text property
  • Only use variants for genuinely mutually exclusive dimensions like variant and size

Everything uses Auto Layout. No absolute positioning inside components. Spacing between elements uses variables from the space/* scale, never typed numbers.

File organization

Non-negotiable for AI compatibility.

Page order inside the DS file:

📖 00 – Cover & Guidelines
🎨 01 – Foundations (color, type, spacing, radius, shadow)
🧩 02 – Components – Primitives
🧩 03 – Components – Composed
📐 04 – Patterns
🖼 05 – Working screens
🗂 99 – Archive
This structure lives in the shared library file. Each of the three site files stays lean — it consumes components published from the library rather than repeating Foundations/Components pages of its own.
Layers are named descriptively. Rectangle 47 and Frame 923 are useless to an AI (and to a developer). Every meaningful layer gets a name: hero-image, cta-primary, card-body.
No detached instances in production frames. If you need to override something, use component properties. Detached instances break the mapping from design to code.

FAQs

Frequently asked questions

1

Does the rebuild need to happen inside the current Figma file, or will it start fresh in a new one?

Decided: each of the three sites moves into its own file with its own access permissions, and the design system becomes a dedicated shared library that all three plug into — built once, updated everywhere. It will be taken care of and prepared for when you join as this is not currently set up.

2

Do you want any visual or style changes made while this gets rebuilt, or just the structural issues fixed?

Structural fixes are the priority for this budget. A visual refresh is a separate conversation we can have once the system itself is solid — it isn't included in this $1,800.

3

Who uses this design system day to day?

There's no dedicated dev team — it's just three of us: the owner, the CTO, and a product strategist (who was previously the designer, and built this system over time before moving into that role). Because of that, this needs to be built with Claude in mind: components and annotations need to be clear enough that Claude can implement directly from the Figma file, not just a human developer.

4

Could this turn into an ongoing collaboration beyond this fixed-scope project?

Possibly. Our product strategist was previously the designer who built this system over time before moving into that role — if this engagement goes well, there's real potential to extend into a longer-term working relationship beyond this initial $1,800 scope.

5

Who's building from this on the development side, and how should it be annotated?

Claude does the implementation — we don't have engineers on staff. Annotate directly in Figma, written clearly enough for Claude to build from directly, rather than a separate Notion page or site.

6

Is there a priority order across the product areas, in case the fixed budget doesn't stretch to cover everything?

Yes: Presentation Website and Employee Web App first, since they're furthest along; Management Dashboard next, since it's currently only wireframe-stage and needs a full rebuild. Your component list, submitted before we start, is what confirms what's realistic within $1,800.

7

Is the $1,800 budget fixed, or can it be extended if scope grows?

It's fixed. $1,800 covers the full engagement end to end — Milestone 1 through Milestone 3 — split as shown in the Rebuild Plan. If the agreed component list turns out to be bigger than expected, we'll prioritise what's covered rather than extend the budget.

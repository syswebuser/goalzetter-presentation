Summary & goal Findings

Scope of work

This covers a full audit and rebuild of our Figma design system, which underpins three product areas: the Presentation Website, the Employee Web App, and the Management Dashboard.

The goal

We want a properly structured, Variable-based design system where every component has a clear property structure, a documented purpose, and a build pattern that scales. Some parts of the file, and some of the visual design itself, could also see a refresh — that's a separate decision from this scope of work, and isn't included in this budget.

What we found

These are general findings from an initial pass, not an exhaustive component-by-component review — that level of detail is what Milestone 1 delivers. Ordered from the most significant issue to things already working well.

Foundational

Styles, not Variables

Colour, spacing, radius, and grid are built with Styles rather than Variables, so there's no single source of truth and no way to support fast, systemic updates.

Structural

Component structure is inconsistent

Some components shape the frame itself directly, adding radius and fill to it; others nest a rectangle inside the frame for the same purpose. Both patterns sit side by side in the same file.

Mixed

Properties are inconsistent, too

Some component families are properly built with Properties, others aren't. Cards are a good example: they only have Variants, no Properties, so using one means checking the different variants and guessing which applies where.

Needs redesign

Some components aren't structurally optimised

Buttons have properly built states and properties, but Primary, Secondary, and Tertiary exist as three separate components rather than a single Property within one Button component. That adds unnecessary duplication, and it's one of the first things to fix when this gets rebuilt.

Untracked

Some colours aren't linked to any style

A number of colour values in the file are hardcoded rather than linked to a Style at all. Even after moving from Styles to Variables, as planned, these need to be found and linked manually first, which makes that migration more work than a straightforward swap.

Partially built

Some components are built but not used within the templates

A few pieces of the system already exist but don't show up anywhere in the actual product screens, which makes those templates less useful in practice.

Needs rebuild

Management Dashboard exists only in wireframe shape

Management Dashboard has an early wireframe pass but no built components yet, so it needs full design and rebuild as part of this scope, not just a refresh. Presentation Website and Employee Web App are further along, though inconsistently: some workflow stages (e.g. Ready for Dev) are populated for one and empty for the other.

By design

The extra numbered pages aren't clutter

What looks like dozens of unlabelled, numbered pages sitting alongside the real structure is intentional: they're a way of adding more pages inside the same file, rather than paying for additional Figma files or seats. Worth knowing when navigating the file, but not something to clean up.

Not a system

The file is generally not annotated

There's no consistent way for components to explain when or how to use them, which makes onboarding, or coming back to the file later, harder than it needs to be.

Good shape

Text styles

In good shape. Most of what was reviewed is correctly linked to text styles.

Strategy

The strategy is:

Design system audit & recommendations — a component-by-component review across all three product areas. Every piece of the system gets categorised: keep as-is, redesign, or build from scratch. Delivered as a written summary with a clear recommendation before any building starts.
Working session on what stays, what's redesigned, and what's gone — walking through the audit findings together, aligning on direction, and confirming the scope of Milestone 2 before it starts.
Build in Figma — the rebuild follows an atomic design structure, working from the smallest unit up: Foundations (Variables, colour, typography, spacing, elevation, grid), Atoms (buttons, inputs, badges, icons), Components (cards, modals, navigation, forms), Templates (page layouts), and Pages (product-ready screens). The file structure reflects this — fewer pages, clearly named, easier to navigate day to day. AI handles the mechanical parts: setting up Variable collections, migrating existing Styles, scaffolding property structures, and batch operations across the file. The design decisions — what each component looks like and how it behaves — are made by hand.

This is our recommended approach. Keeping the current file structure is also an option — it just carries more complexity and is harder to work from as the system grows.

How AI will be used

Given how much restructuring this file needs, AI is used deliberately, and only where it speeds up mechanical work. It is never used in place of design judgment.

AI will

Organise the file — pages, sections, and naming conventions.
Build Variable collections — colour, spacing, radius, shadow/elevation, grid, and migrate existing Styles into them via a direct Figma MCP connection.
Scaffold component properties — boolean, instance-swap, and text properties, once a component's structure has been specified.
Handle batch operations — renaming layers, applying styles/variables consistently across dozens of instances, restructuring pages.

AI will not

Audit the file — reviewing usability, structure, and scalability requires design judgment that AI review can't substitute for.
Decide what's kept, redesigned, or rebuilt — a design and business decision made together with us.
Design or rebuild components visually — given how inconsistent the current patterns are, this is manual design work.
Write annotation content — usage guidance requires understanding our product and use cases.

Budget and cost

A few things drive what actually fits inside this budget.

1

Audit: across three product areas and a dozen-plus component categories, realistically 8-10 hours before any building starts.

2

Foundation: migrating Styles to Variables can be AI-assisted, but every value still needs a human review pass.

3

Component rebuild: roughly 3-4 hours per family (design plus a full Properties structure), multiplied by however many families need it.

4

Annotation: AI can help speed up the process, but each component still needs its use case thought through and written up clearly.

5

Fixed scope: $1,800 covers the whole engagement, audit through rebuild, with no possibility to extend. If Milestone 1 turns up more work than the budget can cover, the component list gets prioritised rather than the budget growing — see the FAQs.

This project is budgeted at $1,800 total, fixed, split across the two milestones above. Milestone 1's findings are what determine what's realistic to fit inside Milestone 2 for that price.

Good to know

FAQs

Frequently asked questions

1

Does the rebuild need to happen inside the current Figma file, or can it start fresh in a new one?

That's something Milestone 1 will help settle properly. As far as we know, there's nothing — no active handoffs, no developer references — tied to the current file that would rule out a clean break, so come into Milestone 1 with a recommendation either way.

2

Is there a live site currently in production?

Yes. Understanding how closely it matches the Figma file is part of Milestone 1, so scope is accurate before the rebuild starts.

3

Do you want any visual or style changes made while this gets rebuilt, or just the structural issues fixed?

Structural fixes are the priority for this budget. A visual refresh is a separate conversation we can have once the system itself is solid — it isn't included in this $1,800.

4

Who uses this design system day to day?

Mainly our own design and product team, plus the developers building from it — so it needs to hold up for both.

5

Who's building from this on the development side, and how should it be annotated?

Our developers work directly off the Figma file today, so annotate in Figma rather than a separate Notion page or site — usage guidance should stay attached to the component itself.

6

Is there a priority order across the product areas, in case the fixed budget doesn't stretch to cover everything?

Yes: Presentation Website and Employee Web App first, since they're furthest along; Management Dashboard next, since it's currently only wireframe-stage and needs a full rebuild. Milestone 1 will confirm what's realistic within $1,800.

7

Is the $1,800 budget fixed, or can it be extended if scope grows?

It's fixed. $1,800 covers the full engagement end to end — Milestone 1 through Milestone 2 — split as shown in the Rebuild Plan. If the agreed component list turns out to be bigger than expected, we'll prioritise what's covered rather than extend the budget.
